# C4 Architecture Model

> **Status: diagram image rendering is descoped.** The HTML/PNG rendering, density QA
> gating, and auto-split for C4 diagrams were provided by the `nano-banana` MCP server,
> which was **removed in #401**. Choosing a replacement rendering engine (options include
> plain Mermaid C4 source, mermaid-cli, Kroki, or Playwright + mermaid.js) is tracked in
> **issue #411**.
>
> Until #411 lands, this command still does the valuable, engine-independent work: it
> analyzes the project and writes a structured **C4 model** (levels, nodes, edges,
> relationships) as Markdown to `docs/architecture/`. It does NOT emit rendered diagram
> images. Do not call any `generate_diagram` / `split_diagram` tool - that engine is gone.

## Arguments

- `LEVEL` (optional): Produce only a specific level - `L1`, `L2`, `L3`, or `L4`. Default: all 4 levels.

## Instructions

When the user invokes `/documentation:c4 [LEVEL]`, analyze the current project and write a C4 architecture model as text.

### Step 1: Analyze the Project

Read the project's key files to understand its architecture:

1. **CLAUDE.md** - Project overview, components, structure
2. **README.md** - High-level description
3. **Directory structure** - `ls` the top-level directories
4. **Key config files** - `pyproject.toml`, `package.json`, `docker-compose.yml`, etc.
5. **Existing docs** - Check `docs/architecture/` for previous C4 outputs

Build a model of:
- Who uses the system (actors/personas)
- What external systems it integrates with
- What containers (apps, DBs, services) make up the system
- What components exist within each container
- What key code elements (classes, modules) implement each component

**C4 Node Types** (use these when describing nodes at each level):

| Node Type | C4 Concept | When to Use |
|-----------|-----------|-------------|
| `person` | Actor / User | External users, admins, operators |
| `system` | External System | Third-party APIs, services not owned by team |
| `system-focus` | System of Interest | The system being documented |
| `container` | Container | Applications, databases, message queues, file stores |
| `component` | Component | Modules, services, controllers within a container |
| `code` | Code Element | Classes, interfaces, functions, data models |

### Step 2: Determine Output Path

```bash
DOCS_DIR="docs/architecture"
mkdir -p "$DOCS_DIR"
```

### Step 3: Write the C4 Model

Write `${DOCS_DIR}/c4-model.md` capturing each requested level as a structured outline.
For each level, list the nodes (id, label, C4 node type, description) and the edges
(source -> target, label). Keep it text-only and version-controllable:

- **L1 System Context** - actors + external systems + the system of interest (5-10 nodes max)
- **L2 Container** - applications, data stores, services within the system boundary; note technology choices
- **L3 Component** - one section per container; 5-15 components each, including edges to other containers
- **L4 Code** - top 3 most architecturally significant components per container; classes/modules within each

Structure `c4-model.md` with one `##` section per level (and `###` per container for L3/L4),
using Markdown tables for nodes and edges. This file is the source of truth a future
rendering engine (#411) can consume.

### Step 4: Report

```
C4 Architecture Model Written

  Output: {DOCS_DIR}/c4-model.md

  L1 Context:   {node_count} nodes, {edge_count} edges
  L2 Container: {node_count} nodes, {edge_count} edges
  L3 Component: {N} containers documented
  L4 Code:      {M} components documented

  NOTE: Diagram IMAGE rendering is descoped pending issue #411. This command
  produced the C4 model as text only.
```
