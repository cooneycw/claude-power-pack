# Grill-yourself transcript — Issue #395

- Date: 2026-06-26
- Issue: #395 — `cpp:update` has no skill drift/orphan detection
- Interviewer: second-opinion MCP (`gemini-3.5-flash`), session `a6203a5c`
- Verdict: **yes-with-caveats**

## Plan grilled

Add skill drift/orphan detection to `/cpp:update`, mirroring the existing MCP
drift architecture (`scripts/drift-detect.sh` + `tests/test_drift_detect.py`):
a backing script classifies generated skills in `~/.claude/skills/` against the
curated `.claude/deprecated-skills.yaml` (shipped by #394/#396), reports a
drift table, and offers user-confirmed per-family removal from a new update.md
step.

## Pre-grill empirical de-risking

Verified that the generated skills' `family:` frontmatter does NOT reliably map
to repo command dirs (`issue` -> `github/`, `agents-md` -> `claude-md/`,
`grill` -> no command dir). So a family->command-dir mapping cannot be used as a
liveness signal — a naive prune would delete live skills (`grill-me`, `issue-*`,
`agents-md-*`, `docs-c4`). This is the exact trap the issue warns about and is
why classification must be curated-list-driven.

## Findings incorporated (caveats)

- **C1 — Python, not bash/awk.** A custom awk YAML parser is fragile (flow
  sequences, inline comments, quotes, indentation). PyYAML 5.4.1 is present in
  base `python3` (and `pyyaml>=6.0` is a dev dep), so the helper is written in
  Python with PyYAML (graceful fallback if import fails). JSON output and
  path-traversal guards are also far safer in Python.
- **C2 — "Partial-family" false positive (review Bug A).** Treating any
  unlisted member of a deprecated family as ORPHANED breaks if a family is only
  *partially* retired (one dead skill among live ones). Fix: gate ORPHANED
  behind an explicit `entire_family: true` flag on the family entry. Without it,
  unlisted members stay **OK**. `wiki` is fully retired, so it gets the flag;
  all three wiki skills are explicitly listed, so none is ORPHANED today.
- **C3 — Destructive prune.** `--prune` moves skills to
  `~/.claude/.backup/skills/<ts>/` instead of `rm -rf`; hard-refuses any skill
  classified OK/IGNORED, refuses names containing `/` or `..`, and verifies the
  resolved target is a direct child of the skills dir.
- **C4 — Hand-authored skills.** Dirs lacking the
  `<!-- GENERATED by claude-power-pack -->` marker are classified IGNORED and
  are never pruned.
- **C5 — Silent-drift limitation.** With no `manifests/` of record, a skill
  removed from CPP without a deprecation-list entry will not be flagged. The
  report and docs state this and direct maintainers to add removed skills to
  `.claude/deprecated-skills.yaml`.

## Rejected / out of scope

- Reviewer suggested restoring a forward manifest-of-record to detect orphans
  precisely. The issue explicitly chose the curated-list approach as primary
  ("lowest risk, ties to #394"); confirmed there is no current forward
  source-of-truth (`codex-skill-gen.py` emits a different `source:` format than
  the installed `manifests/*` skills). Deferred; documented as C5.
- Windows portability (reviewer): CPP is Docker/Linux-centric (bash hooks,
  systemd, Makefile); not a constraint. Python still chosen for robustness.
