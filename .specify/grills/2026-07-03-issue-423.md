# Grill-yourself: issue #423 (migrate /qa:test to upstream playwright-mcp, retire the fork)

**Date:** 2026-07-03. **Trigger:** >200-line/>5-file change touching `docker-compose.yml`,
`.woodpecker.yml`, and heavily-coupled tests. **Method:** inline self-interrogation
(second-opinion MCP grill not used; deep local context available from spike #419).

**Reframe decision (reviewer-approved):** proceed single-session, drop the stale `#421`
dependency and "wrapper verbs", fold Task 2 into docs. Registration path **A (npx/stdio)**.
Reviewer caveat: **CPP must still facilitate the install** (the `/cpp:init` browser step
stays; it registers upstream via npx instead of building the fork).

## Q1: What is the highest-risk part?
Test coupling. `tests/test_docker_compose_config.py` asserts the playwright service exists,
its port mapping, the `browser` profile in the build/config/smoke commands, image-security
`when.path`, the drift port map (8081), the src-file map, and smoke paths. Every one must be
updated to the new reality (no browser service, no `browser` profile member, no playwright in
the CVE-scan image list). Mitigation: `make test` is the gate; update asserts to match, do not
just delete them blindly.

## Q2: Any invariant I could silently break in CI config?
Yes - the Woodpecker `when.path` superset invariant (#384): `runtime-smoke` `depends_on`
`image-security`, so image-security's path list must stay a SUPERSET of runtime-smoke's.
Removing `mcp-playwright-persistent/**` from BOTH keeps them equal - safe. Removing from only
one would error the pipeline. Action: edit both lists together.

## Q3: Is `browser` profile removal safe and complete?
`mcp-playwright-persistent` is the SOLE service with `profiles: ["browser"]`. Removing it
empties the profile. To avoid dead references I remove `--profile browser` end-to-end:
docker-compose header comments, `.woodpecker.yml` build line, `scripts/runtime-smoke.sh`,
and the `make docker-refresh/health PROFILE="core browser"` -> `PROFILE="core"` in docs and
`/cpp:init`. Inconsistency between a test assert and the actual command FAILS the test, so
these move together.

## Q4: Does the reviewer caveat (CPP still facilitates install) change the design?
Yes. `/cpp:init` keeps a browser-automation step, but it swaps
`claude mcp add playwright-persistent --transport sse --url http://127.0.0.1:8081/sse` for
`claude mcp add --transport stdio --scope user playwright -- npx -y @playwright/mcp@latest --headless`,
guarded by an `npx` presence check (mirroring the existing sequential-thinking extra). It is now
Docker-independent, so its registration must not sit behind the Tier-3 Docker gate.

## Q5: Are the upstream tool names I will write into /qa:test correct?
Taken from the spike's verified v0.0.77 mapping table (not memory): browser_navigate,
browser_click, browser_type, browser_fill_form, browser_select_option, browser_hover,
browser_tabs, browser_navigate_back, browser_take_screenshot, browser_snapshot,
browser_pdf_save (behind `--caps pdf`), browser_evaluate, browser_wait_for,
browser_console_messages. No `session_id` param (implicit single context). The 6 CPP
conveniences (go_forward, reload, get_text, get_attribute, get_content, query_selector_all)
