## Q1. Will adding fields + new branches break the 15 existing Woodpecker tests?

Risk: `generate_woodpecker()` currently emits a byte-exact naive pipeline that
tests assert against. Adding stages or `depends_on` unconditionally would break them.

Resolution: ALL new behavior is opt-in via `WoodpeckerConfig` flags defaulting
to `False`. The naive path (no flags set) must remain byte-identical - no
`depends_on`, no reordering, no new stages. Verify by running the existing
`TestWoodpecker` / `TestWoodpeckerAWSSecrets` classes unchanged and green.

## Q2. Can a generic generator faithfully reproduce CPP's image-security + smoke apparatus?

Risk: CPP's real `.woodpecker.yml` image-security/runtime-smoke stages use the
docker-buildx plugin, privileged mode, docker-socket mounts, and bespoke shell
scripts (runtime-smoke.sh with LocalStack + fake agent). Inlining that generically
would emit broken/unusable YAML for an arbitrary project.

Resolution (altitude decision): the generator emits REAL, self-contained steps
where they are project-agnostic:
  - secret-scan  -> `zricethezav/gitleaks` detect (self-contained, always valid)
  - image-security -> hadolint over Dockerfiles + `aquasec/trivy` fs/config scan
    (no image build required, so no docker socket needed)
  - runtime-smoke -> calls a `make smoke` target (project-owned), NOT CPP's
    bespoke LocalStack rig
The full build-CVE + LocalStack apparatus is DOCUMENTED in the knowledge doc,
not code-generated. Command doc + report must not over-promise. This keeps
generated output runnable and honest.

## Q3. Secrets-bootstrap generalization - over-engineering risk?

Risk: turning the hardcoded `woodpecker/bootstrap-secrets.py` (secret=essent-ai)
into a config-driven generator could balloon.

Resolution: ship a parametrized `.example` template with argparse for
`--secret-name` / `--region` and clearly-marked placeholder keys. Lean, copyable,
no new runtime code path in lib/cicd.

## Q4. New command drift - does /cicd:woodpecker need registration elsewhere?

Commands are hand-edited .md files (no central manifest). But surfaces that list
commands will drift if not updated: `.claude/commands/cicd/help.md`, `CLAUDE.md`
CI/CD section. Must update BOTH plus fix `pipeline.md` description.

## Q5. Will `make verify` (lint + test + typecheck + mypy) pass?

- mypy: match existing type annotations in pipeline.py/config.py exactly.
- em-dash rule (CLAUDE.md core directive): knowledge doc + command doc must use
  single dashes only, never U+2014 / U+2013.
- new tests are pure string-codegen (no git subprocess), so the "validate
  container has no git" CI trap (memory) does not apply.

## Q6. Path collisions?

`templates/woodpecker/` (server/agent INSTALL + secrets bootstrap) is distinct
from existing `templates/workflows/woodpecker-*.yml` (PIPELINE templates) and the
repo's own `woodpecker/` dir. No collision; keep the separation-of-concerns clear
and cross-reference rather than duplicate.

- C2: generated hardening stages are self-contained where possible, honest
